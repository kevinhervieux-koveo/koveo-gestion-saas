## 🔴 HIGH (2)

### W77 — `/api/organizations/<id>/buildings` retourne 0 rangée pour chaque org
- **Surface :** Web/API
- **Rôles touchés :** super_admin (vérifié), manager + admin (probable)
- **Symptôme :** 5 pages manager affichent « Aucun élément trouvé » après le choix d'org :
  - `/manager/bills`
  - `/manager/budget`
  - `/manager/common-spaces-stats`
  - `/manager/maintenance/inventory`
  - `/manager/maintenance/projects`
- **Cause racine :** `GET /api/organizations/<orgId>/buildings` retourne 200 avec `[]` pour Demo, 563, Koveo, MCP-1 et MCP-2 — tandis que `/api/buildings?organizationId=<id>` retourne les bonnes données.
- **Régression de :** Pass #25 W43 (même classe d'impact, endpoint différent).
- **Fix proposé :** repointer les pages cassées vers `/api/buildings?organizationId=<id>` OU corriger le scope-resolver de la route per-org.

### W78 — `/admin/performance` reste figé sur « Loading... »
- **Surface :** Web
- **Rôles touchés :** super_admin (vérifié), admin (probable)
- **Symptôme :** Page jamais rendue. Aucun `<main>`, aucun `<h1>`, aucun message d'erreur visible. Titre du browser tombe sur le tagline marketing.
- **Cause racine :** `TypeError: Cannot convert object to primitive value` lancé 6× durant le rendu React, attrapé par l'error boundary qui re-monte et re-bondit dans la même erreur. Les fetch API réussissent (auth/user, impersonation-status, users/me/buildings, onboarding/me, performance/web-vitals — tous 200) — la page plante avant même d'aller chercher ses données.
- **Indice :** typique d'un objet rendu directement comme enfant JSX (`{someObj}` au lieu de `{someObj.field}`).
- **Fix proposé :** localiser le node JSX qui rend un objet ; ajouter un error boundary visible pour éviter le « Loading... » infini.

### W81 — Routes super_admin nommées sous `/admin/...` au lieu de `/super_admin/...`
- **Surface :** Web (architecture d'information / taxonomie de routes)
- **Rôles touchés :** super_admin, admin (clarté de scope)
- **Symptôme :** Plusieurs pages dans la section `/admin/...` sont en réalité **super_admin-only** par leur fonctionnalité (gestion d'orgs, audit logs, impersonation, qualité, compliance, performance, bulk import, KPI). Le nommage `/admin/...` suggère qu'un admin régulier peut y accéder, alors qu'une grande partie est gated à super_admin uniquement (création d'org = super_admin par règle canonique).
- **Exemples concrets à renommer :**
  - `/admin/organizations` → `/super_admin/organizations` (création d'org = super_admin only)
  - `/admin/impersonation-log` → `/super_admin/impersonation-log` (audit d'usurpation système)
  - `/admin/kpi-dashboard` → `/super_admin/kpi-dashboard` (télémétrie produit)
  - `/admin/quality` → `/super_admin/quality` (métriques internes de build)
  - `/admin/compliance` → `/super_admin/compliance` (analyse Loi 25)
  - `/admin/performance` → `/super_admin/performance` (perf interne)
  - `/admin/org-access` → `/super_admin/org-access` (accès cross-org)
  - `/admin/bulk-document-import` → `/super_admin/bulk-document-import` (admin only)
  - `/admin/permissions` → `/super_admin/permissions` (RBAC matrix gestion)
  - `/admin/document-tags` → `/super_admin/document-tags` (Koveo system tags)
- **Pourquoi ça compte :**
  - Reflète fidèlement la hiérarchie canonique des rôles (`super_admin > admin > manager > resident > tenant`).
  - Évite la confusion pour un admin régulier qui voit des liens `/admin/...` dans le sidebar mais reçoit 403 sur la moitié.
  - Aide le routing / role-guard middleware à appliquer le bon check par préfixe d'URL.
  - Aligne le nommage avec le pattern existant `/manager/...` qui est manager+admin+ (pas confondu avec `/admin/...`).
- **Fix proposé :**
  1. Identifier chaque page `/admin/...` qui est super_admin-only (vs vraie admin+).
  2. Créer les nouvelles routes `/super_admin/...` avec le même composant.
  3. Garder `/admin/...` comme alias avec redirection 301 vers `/super_admin/...` pendant 1–2 cycles pour préserver les bookmarks.
  4. Mettre à jour le sidebar nav et les breadcrumbs.
  5. Éventuellement déplacer les pages réellement admin+ (genre `/admin/document-tags` si admin peut aussi y toucher) sous le bon préfixe selon leur scope réel.
- **Décision préalable nécessaire :** valider page-par-page lesquelles sont vraiment super_admin-only vs admin+ avant de bouger les routes.
